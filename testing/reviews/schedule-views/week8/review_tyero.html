1. Code Follows Design and Implementation Conventions

The code follows the proper format and indentation conventions. Variable naming also follows a logical convention (adding 'm' to the beginning of member variables). The view classes however, don't append "UI" to the name but rather "View". For instance, "CalendarTableView". I'm ok with this. I'm not sure how I feel about "Dlg" instead of "Dialog" for the "EditScheduleItemDlg".




2. Javadoc Is Complete and Follows Design and Implementation Conventions

Javadocs are incomplete. For the methods that they are done with they seem fine.




3. Design of the Code is Good

The code follows a MVC pattern with ScheduleEditWidget acting as the controller for CalendarTableView, CalendarListView and EditScheduleItemDlg. All of the business logic takes place in the model or controller. This seems reasonable. There are extra unwritten methods for the filter functionality...but the filter button doesn't display on the screen so this isn't a problem.




4. Code Passes Its Tests and Has Sensible User-Level Behavior

I've said it before and I'll say it again, these are the most reliable parts of the whole scheduler application. Drag and drop works fine and the list displays correctly. The view is snappy upon switching between calendar and list.
